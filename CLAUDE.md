# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`ir-webcmp` — a Stencil 4 library of web components powering the IglooRooms PMS UI. Components are framework-agnostic custom elements consumed by host HTML pages (one per feature, see `src/pages/index-*.html`). The backend is `https://gateway.igloorooms.com/IR`.

## Feature notes (`.claude/memory/`)

Before touching a specific feature, **read the matching note** in `.claude/memory/` — start with `.claude/memory/INDEX.md`. Each `feature_*.md` documents that feature's child component map, services/stores, naming gotchas, and conventions that aren't obvious from the code alone. `patterns_report_pages.md` is the template for new reports. `screenshots_folder.md` maps the `screenshots/` PNGs to components (including two name mismatches: "Daily Occupancy" → `ir-monthly-bookings-report`, "Availability Alert" → `ir-unbookable-rooms`).

## Commands

- `npm start` — dev server on **port 7594** with watch (`stencil build --dev --watch --serve -p 7594`). Playwright's `baseURL` is also 7594, so keep this port.
- `npm run build` — production build with `--docs` (regenerates `readme.md` files next to each component via the `docs-readme` output target).
- `npm test` — Playwright e2e against `tests/*.spec.ts`. The dev server is **not** auto-started by Playwright; start it manually first.
- `npm run test:ui` — Playwright UI mode.
- `npm run test.watch` — Stencil's own spec/e2e watcher (`stencil test --spec --e2e --watchAll`). Use this for `src/components/**/test/*.spec.tsx`.
- `npm run generate` — `stencil generate` scaffolds a new component (tag, css, spec).
- Run a single Playwright test: `npx playwright test tests/booking.spec.ts -g "name"`.
- Run a single Stencil spec: `npx stencil test --spec --no-watch <path-to-spec>`.

## Architecture

### Output targets (stencil.config.ts)
Four targets are emitted: `dist` (with loader), `dist-custom-elements`, `docs-readme`, and `www`. Consumers can load either the lazy ESM bundle (`/build/ir-webcmp.esm.js`) or import custom elements directly. `src/global/app.ts` is the global script and **must register every WebAwesome component used anywhere in the library** via side-effect imports — if a component uses a new `<wa-*>` element, add the import there or it won't be bundled.

### Path aliases (tsconfig.json)
- `@/*` → `src/*`
- `@components/*` → `src/components/*`

Use these in imports rather than long relative paths (the codebase mixes both, but the alias form is the direction it's moving).

### Top-level src/ layout
- `components/` — all Stencil components, grouped by feature folder (each component has its own folder containing `*.tsx`, `*.css`, optional `test/`, optional `readme.md` generated by `docs-readme`).
- `services/` — class-based API clients (one class per domain). They use `axios` directly and rely on the global axios config set by `Token`.
- `stores/` — `@stencil/store`-based shared state. Each store exports `state`, `onChange`, plus named setter functions.
- `models/` — DTOs, interfaces, and some Zod schemas (`*Schema`).
- `decorators/` — custom TS decorators: `@Debounce`, `@ClickOutside`, `@OverflowLock`.
- `utils/`, `common/` — pure helpers, shared CSS (`common/table.css`, `common/sheet.css`).
- `pages/index-<feature>.html` — one HTML harness per top-level component, served by the dev server.
- `global/app.ts` — global script (WebAwesome registrations + theme CSS).
- `global/app.css` — global stylesheet emitted to all targets.

### Component conventions
- **`scoped: true`** is used project-wide. Shadow DOM is **not** used. Styles can therefore leak in/out — be deliberate with class names.
- Component tags are namespaced by area: `ir-*` (default), `igl-*` (everything under `igloo-calendar/`), and `ac-*`, `ota-*`, `requirement-check` for a few specials. Match the prefix of the folder you're working in.
- Props are declared with `@Prop()` and **JSDoc-documented** — these comments feed `docs-readme`. New props should have JSDoc.
- Events use `@Event()` with camelCase names (`requestPageChange`, `checkInRoom`). Handlers consistently call `e.stopImmediatePropagation(); e.stopPropagation();` before mutating state — follow that pattern.
- Page-level components accept these props by convention: `ticket` (auth token), `propertyid` (number), `p` (property alias / `aname`), `language` (`'en'` default). Init flow: `componentWillLoad` → `tokenService.setToken(ticket)` → `init()` → `Promise.all([...])` for parallel fetches → set `isPageLoading = false` in `finally`.
- Page-level components typically also render `<ir-toast>` and `<ir-interceptor handledEndpoints={[...]} />` siblings, and gate first render on `<ir-loading-screen>`.
- Multiple stylesheets: use `styleUrls: ['x.css', '../../../common/table.css']` when pulling in shared CSS; otherwise `styleUrl: 'x.css'`.

### HTTP / auth
- **All HTTP goes through axios with `Token` setting the base URL and Authorization header.** `new Token().setToken(ticket)` installs a request interceptor (once, statically) and points `axios.defaults.baseURL` at the gateway. Services do not configure axios themselves.
- API response convention: every backend call returns `{ ExceptionMsg, My_Result, ... }`. Services check `data.ExceptionMsg !== ''` and `throw new Error(data.ExceptionMsg)`, then return `data.My_Result`. Mirror this pattern for new endpoints.
- Many service methods validate input with Zod: `const payload = SomePropsSchema.parse(props); await axios.post('/Endpoint', payload);`. Add a matching `*PropsSchema` in the service's `types.ts` when introducing a new request shape.
- Services are instantiated per component (`private bookingService = new BookingService();`) — they hold no instance state, so this is cheap.

### Stores (`@stencil/store`)
Pattern (see `src/stores/arrivals.store.ts` for a canonical example):
1. Define a typed `initialState`.
2. `export const { state: <name>Store, onChange: on<Name>StoreChange } = createStore(initialState);`
3. Export named mutator functions (`set<X>`, `initialize<X>`) — components import these rather than mutating the store object directly.
4. Run derived-state pipelines inside mutators (filtering, pagination calculations) so the store stays internally consistent.

When components need to react to store changes, they subscribe in `componentWillLoad` with `on<Name>StoreChange('field', cb)`.

### Per-feature requirements
The repo's `requirements.md` is a deep-dive spec for a single component (`igl-to-be-assigned`). Treat it as the model for documenting any non-trivial new component: purpose, inputs/deps, state, lifecycle, interaction flows, edge cases. Don't generalise it into a "rules" doc — it's a per-component artifact.

### Tests
- **Stencil unit specs**: `src/components/<x>/test/<x>.spec.tsx` using `newSpecPage` from `@stencil/core/testing`. Most are smoke renders today.
- **Playwright e2e**: `tests/*.spec.ts` against `http://localhost:7594`. Helpers in `tests/utils.ts`; fixture data in `tests/data/`. Dev server must be running.

### Building a new component — checklist
1. Run `npm run generate` (or copy a sibling folder) under the appropriate feature folder. Match the tag prefix of that folder.
2. Add `scoped: true`, JSDoc each `@Prop`, declare events with `@Event()`.
3. If it's a page-level component: take `ticket` / `propertyid` / `language`, install token via `Token`, render `<ir-toast>` + `<ir-interceptor>`, gate on `<ir-loading-screen>`.
4. If it talks to the backend: add the method to the matching service in `src/services/`, define a Zod `*PropsSchema` in the service's `types.ts`, return `data.My_Result`.
5. If it needs shared state: add a store in `src/stores/` following the `arrivals.store.ts` pattern.
6. If it uses a new WebAwesome element: register it in `src/global/app.ts`.
7. If it needs a demo page: add `src/pages/index-<feature>.html` mirroring an existing one (token bootstrap via `src/scripts/index.js`).
8. Add a spec under `test/`; consider a Playwright spec in `tests/` if it's a top-level page.
